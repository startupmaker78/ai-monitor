# Вебмонитор — Roadmap MVP

> Каждый этап заканчивается чем-то рабочим, что можно показать пользователю и получить фидбек.
> Порядок выбран так, чтобы быстрее дойти до первой реальной ценности.

---

## Параллельный касдев (с недели 2, параллельно с разработкой)

> Касдев идёт **параллельно** с разработкой начиная с момента появления демо-стенда (конец Этапа 1). Не ждём готового MVP.

**Цель**: получить 3–5 платящих клиентов (оплата первого месяца) до полной готовности MVP — как подтверждение спроса.

**Аудитория**: 10–20 владельцев сайтов малого бизнеса из личного окружения Лёши.

**Процесс**:
- Показываем демо-стенд (публичная ссылка, без регистрации)
- Собираем обратную связь: что понятно, что непонятно, за что готовы платить
- Подключаем желающих как первых клиентов — они используют продукт по мере его готовности
- Фиксируем инсайты, корректируем приоритеты разработки

**Критерий успеха**: 3–5 клиентов оплатили первый месяц → спрос подтверждён.

**Следующий шаг после подтверждения**: идём к дизайнерам nolim.cc с готовой механикой продаж (кейсы, цифры, рабочий продукт).

## Этап 1 — Регистрация подрядчика, инвайт-система и демо-стенд (неделя 2)
*Цель: подрядчик может добавить первого клиента, новый пользователь сразу видит ценность продукта*

- [ ] Страница регистрации подрядчика
- [ ] Кабинет подрядчика: список клиентов
- [ ] Форма добавления клиента: URL сайта, название компании, email, телефон, ID Яндекс.Метрики
- [ ] Генерация InviteToken + отправка email через Resend (временный пароль)
- [ ] Страница для владельца по invite-ссылке: смена пароля при первом входе
- [ ] После принятия инвайта: владелец видит пустой дашборд
- [ ] Связь Owner → Contractor в БД
- [ ] **Демо-стенд**: при регистрации подрядчика автоматически создаётся демо-клиент с заполненными данными
- [ ] Публичная ссылка на демо (без авторизации): кнопка «Скопировать ссылку на демо»

**Результат для показа**: подрядчик регистрируется → сразу видит демо с живым дашборд → приглашает реального клиента → клиент принимает приглашение.

- [ ] Настройка rrweb как зависимости
- [ ] Сборка `tracker.js` (esbuild): загрузка по `?token=`, минимальный размер

- [ ] Cron-задача: удаление сессий старше 30 дней
- [ ] Инструкция по установке для Тильды (скриншоты, шаг за шагом)

**Результат для показа**: вставляем скрипт в тестовый сайт Тильды, открываем несколько страниц с разных устройств — видим сессии в дашборде.

## Этап 3 — Яндекс.Метрика + ежедневные снапшоты (неделя 4)
*Цель: дашборд показывает реальные метрики; фундамент для будущей аналитики периодов*

- [ ] Страница настроек: ввод ID счётчика и API-токена Метрики
- [ ] `GET /api/metrika/sync` — загрузка метрик за 30 дней
- [ ] **Ежедневный снапшот MetricsSnapshot**: visits, uniqueVisitors, conversions, bounceRate, avgSessionDuration, goals (JSON)
- [ ] Vercel Cron: ежедневная синхронизация (00:30 МСК)
- [ ] Виджеты на дашборде: визиты, конверсия, отказы, время на сайте
- [ ] Графики по дням (recharts)
- [ ] Кнопка «Синхронизировать сейчас»

**Результат для показа**: дашборд показывает реальную статистику сайта с графиком за 30 дней.

## Этап 4 — AI-анализ и статусы рекомендаций (неделя 5)
*Цель: кнопка «Анализировать» выдаёт рекомендации; пользователь управляет их статусами*

- [ ] Дизайн промпта для Claude (итерируем до хорошего качества)
- [ ] `POST /api/analysis/run` — запуск анализа, проверка месячного лимита
- [ ] Фоновая обработка: сбор данных (сессии + MetricsSnapshot), вызов Claude API
- [ ] Парсинг ответа → массив рекомендаций любой длины, сохраняем все
- [ ] Страница «Рекомендации»: показываем топ-10 по умолчанию, кнопка «Показать ещё N»
- [ ] Карточки по приоритетам (Критично / Важно / Хорошо)
- [ ] **Статусы рекомендаций**: кнопки «Принять» (→ IN_PROGRESS), «Выполнено» (→ DONE), «Отказаться» (→ REJECTED + поле для причины)
- [ ] Фильтры: по приоритету, по статусу
- [ ] История анализов: список прошлых запусков, можно открыть
- [ ] Счётчик «Осталось анализов в этом месяце: N»
- [ ] Индикатор прогресса пока анализ идёт (~30 сек)

**Результат для показа**: запускаем анализ, видим 10+ рекомендаций, принимаем одну в работу, от другой отказываемся с причиной, одну отмечаем выполненной.

## Этап 5 — Аналитика динамики: сравнение по произвольным периодам (неделя 6–7)
*Цель: владелец видит как изменились метрики за выбранный период*

- [ ] Компонент выбора периода на дашборде:
  - «С последних правок» (автоматически — дата последней DONE-рекомендации)
  - «С выбранной даты» (date picker)
  - «За последний месяц / 3 месяца / год»
  - «Между двумя произвольными датами»
- [ ] API: `GET /api/metrics/compare?from=…&to=…` — сравниваем два снапшота из MetricsSnapshot
- [ ] Виджеты сравнения: текущее значение + дельта (↑ +12% / ↓ -3%)
- [ ] Граф с двумя выделенными точками (начало и конец периода)
- [ ] Обработка ситуации «данных за эту дату нет» (берём ближайший снапшот)

**Результат для показа**: выбираем «За последний месяц» — видим какие метрики выросли, какие упали, на сколько процентов.

## Этап 6 — Аналитика по каждой рекомендации (неделя 7–8)
*Цель: видно какая правка дала какой результат*

- [ ] При переводе рекомендации в IN_PROGRESS: фиксируем `metricsBefore` (текущий MetricsSnapshot)
- [ ] Cron через 7 дней после DONE: фиксируем `metricsAfter`
- [ ] Мини-блок «до/после» на карточке выполненной рекомендации
- [ ] Страница «Эффект по рекомендациям»: таблица всех DONE-рекомендаций, сортировка по наибольшему росту ключевой метрики
- [ ] Суммарный блок: «Благодаря X рекомендациям конверсия выросла в среднем на Y%»

**Результат для показа**: видим конкретную рекомендацию и рядом — конверсия была 2.1%, стала 3.4%.

## Этап 7 — Тарифы и оплата (неделя 8–9)
*Цель: можно взять деньги с клиента*

- [ ] Страница тарифов: ежемесячные и годовые цены, выделен Pro
- [ ] Enterprise-тариф: форма запроса (email + описание)
- [ ] Интеграция ЮКасса через ООО «Супер Просто»: создание подписки, страница оплаты
- [ ] Поддержка месячной и годовой подписки
- [ ] Webhook: обработка успешной оплаты, сброс счётчика анализов
- [ ] Активация тарифа после оплаты
- [ ] Ограничение кол-ва анализов по тарифу (месячный лимит)
- [ ] Ограничение кол-ва посетителей по тарифу (счётчик в Session)
- [ ] Страница «Подписка» в настройках: текущий план, дата следующего списания, смена плана
- [ ] Email-уведомление об оплате (владелец + подрядчик)

**Результат для показа**: клиент выбирает Pro-тариф, оплачивает месячную или годовую подписку, тариф активируется, лимиты применяются.

## Этап 8 — Комиссия и кабинет подрядчика (неделя 9–10)

- [ ] Начисление 50% комиссии при каждой оплате клиента
- [ ] Страница «Заработок»: таблица — клиент, тариф, дата оплаты, сумма, комиссия
- [ ] Итого к выводу

**Результат для показа**: подрядчик видит сколько заработал на каждом клиенте и может запросить вывод.

## Этап 9 — Полировка и запуск (неделя 10–11)
*Цель: продукт готов к масштабированию через партнёров-дизайнеров*

> К этому этапу у нас уже есть 3–5 платящих клиентов из касдева и живые кейсы — используем их в маркетинге.

### Технические задачи
- [ ] Онбординг: подсказки при первом входе
- [ ] Пустые состояния (empty states) на всех страницах
- [ ] Мобильная адаптация дашборда
- [ ] Страница с инструкцией по установке скрипта (Тильда)
- [ ] Политика конфиденциальности + пользовательское соглашение
- [ ] Мониторинг ошибок (Sentry)
- [ ] Нагрузочное тестирование трекинг-скрипта
- [ ] Настройка домена вебмонитор.рф: DNS, SSL-сертификат, Punycode-URL для скрипта
- [ ] Верификация домена в Resend (@вебмонитор.рф)

### Маркетинговые материалы
- [ ] Лендинг вебмонитор.рф (на Тильде): описание продукта, тарифы, демо-ссылка
- [ ] 2–3 кейса первых платящих клиентов из касдева: что делали, какой результат по метрикам
- [ ] Страница для партнёров-дизайнеров: как работает реферальная механика, сколько можно заработать

### Запуск через nolim.cc
- [ ] Подготовка партнёрского питча: 1 страница — ценность для дизайнера, механика 50%, пример заработка
- [ ] Email-рассылка по базе nolim.cc (~4 500 чел.) с приглашением стать партнёром
- [ ] Личный обзвон топ-20 наиболее активных дизайнеров из базы
- [ ] Онбординг первых 10–20 дизайнеров-партнёров: регистрация, демо-стенд, первый клиент
- [ ] Сбор отзывов и кейсов от первых партнёров и их клиентов

**Результат**: 10–20 дизайнеров-партнёров подключены, у каждого есть как минимум 1 клиент в воронке.

## После MVP (бэклог)

- Воспроизведение сессий (rrweb player) в интерфейсе
- Тепловые карты кликов
- A/B тестирование
- Нотификации (Telegram-бот для подрядчика)
- API для интеграций
- Инструкции по установке скрипта для WordPress, Битрикс, кастомных сайтов
- OAuth для Яндекс.Метрики (вместо ручного API-ключа)
- Автоматический вывод средств подрядчикам через СБП / ЮКасса Payouts
- Enterprise-онбординг
- **Аналитика по подрядчикам для nolim.cc**: суперадмин-панель или экспорт — кто из дизайнеров сколько клиентов подключил, сколько заработал, кто самый активный партнёр. Рейтинг партнёров.
