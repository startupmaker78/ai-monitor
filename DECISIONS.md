## 2026-04-22 — Название продукта: Вебмонитор

**Решение**: продукт называется **Вебмонитор** (по-русски, единое написание везде).

**Причина**: кириллическое название органично для российского рынка, хорошо читается, совпадает с доменом вебмонитор.рф. Ранее рабочее название «AI Monitor» — отменено.

## 2026-04-22 — Домен: вебмонитор.рф (кириллический IDN)

**Решение**: использовать домен **вебмонитор.рф** для всего — основной сайт, API, скрипт трекинга, email.

**Что нужно учесть**:
- Кириллические домены передаются в ASCII через Punycode. Получить точную форму: `node -e "console.log(require('punycode').toASCII('вебмонитор.рф'))"`
- В теге `<script src="...">` в Тильде использовать Punycode-URL (Тильда может не поддерживать IDN)
- В коде Next.js для `NEXTAUTH_URL` и конфигурации Resend — Punycode-форма
- DNS-записи на регистраторе заполняются в Punycode

**Статус**: домен куплен.

## 2026-04-22 — Выбор фреймворка: Next.js 14

**Решение**: использовать Next.js 14 с App Router как единый фреймворк для фронта и бэка.

**Причина**: 
- Один репозиторий, один деплой на Vercel — меньше сложности для MVP
- Лёша не программист, один стек проще поддерживать
- App Router + Server Actions упрощают работу с данными
- Vercel нативно поддерживает Next.js (Cron Jobs, Background Functions)

**Альтернативы, которые отклонили**:
- Отдельный React-фронт + Express/Fastify бэк — лишняя сложность для MVP
- Remix — хорош, но меньше экосистема и меньше туториалов

## 2026-04-22 — База данных: Yandex Cloud Managed PostgreSQL + Prisma

**Решение**: Yandex Cloud Managed PostgreSQL с первого дня, Prisma как ORM.

**Причина**:
- Нет смысла начинать на Neon и потом мигрировать — миграция под первыми клиентами создаёт риск
- Yandex Cloud даёт managed-сервис: автобэкапы, failover, connection pooler (PgBouncer встроен)
- Prisma работает с любым PostgreSQL, меняется только `DATABASE_URL`
- Регион ru-central1 (Москва) — 152-ФЗ соблюдается с первого дня

**Стоимость**: Yandex Managed PostgreSQL от ~$15-25/мес для минимального кластера (1 хост, 2 vCPU, 8 GB RAM).

**ОТМЕНЕНО**: ~~Neon PostgreSQL для разработки~~ — миграция нежелательна, стартуем сразу на Yandex Cloud.

## 2026-04-22 — Российский хостинг для данных: Yandex Cloud

**Решение**: для хранения данных на продакшне использовать **Yandex Cloud** (Managed PostgreSQL + Object Storage).

**Сравнение вариантов**:

| Провайдер     | Managed PostgreSQL | Object Storage (S3) | CDN | Оценка     |
|---------------|--------------------|---------------------|-----|------------|
| Yandex Cloud  | ✅ Полноценный      | ✅ S3-совместимый    | ✅  | **Выбран** |
| Selectel      | ✅ Есть             | ✅ S3-совместимый    | ✅  | Запасной   |
| Timeweb Cloud | ✅ Есть             | ✅ Есть              | ⚠️  | Дешевле    |
| Cloud.ru      | ✅ Есть             | ✅ Есть              | ✅  | Дороже     |
| Рег.ру        | ⚠️ Базовый          | ❌ Нет               | ❌  | Не подходит|

**Причина выбора Yandex Cloud**:
- Managed PostgreSQL: автобэкапы, point-in-time recovery, failover — не нужно администрировать вручную
- Object Storage: S3-совместимый API — код меняется минимально (только endpoint и credentials)
- CDN — важен для раздачи tracker.js на каждый просмотр страницы
- Хорошая документация, широкое сообщество в РФ

**Архитектура данных (продакшн)**:
- Compute: Next.js на Vercel (без хранения PD, допустимо в EU)
- БД: Yandex Cloud Managed PostgreSQL (регион ru-central1, Москва)
- Файлы: Yandex Object Storage (регион ru-central1)

**Запасной вариант**: Selectel — аналогичные возможности, чуть дешевле, менее зрелая документация.

## 2026-04-22 — 152-ФЗ: Yandex Cloud с первого дня

**ОТМЕНЕНО**: ~~двухфазный подход (Neon на разработке → миграция перед запуском)~~

**Новое решение**: Yandex Cloud (Managed PostgreSQL + Object Storage, регион ru-central1) используется с первого коммита.

**Причина**:
- Миграция под первыми клиентами — ненужный риск для продукта
- Экономия на Neon/R2 не покрывает стоимость простоя при миграции
- Yandex Cloud имеет бесплатный грант для новых аккаунтов — первые месяцы разработки бесплатны

**Vercel + Yandex Cloud совместимость**: Vercel подключается к внешним PostgreSQL через стандартный TCP (SSL). Это стандартная практика — аналогично Vercel + AWS RDS. Yandex Object Storage имеет S3-совместимый API, подключается через `@aws-sdk/client-s3` с кастомным endpoint `storage.yandexcloud.net`.

**Статус**: ЗАКРЫТ, переходим на Yandex Cloud с первого дня.

## 2026-04-22 — Хранение rrweb-событий: Yandex Object Storage

**Решение**: сырые rrweb-события хранить в Yandex Object Storage (регион ru-central1).

**Причина**:
- Rrweb-события могут быть очень объёмными (мегабайты на сессию)
- Хранить большие JSON-блобы в PostgreSQL дорого и медленно
- Yandex Object Storage: S3-совместимый API (подключаем через `@aws-sdk/client-s3`), регион РФ, низкая цена (~₽1-2 за GB/мес)
- Lifecycle policy: автоудаление объектов старше 30 дней настраивается нативно

## 2026-04-22 — Деплой — Yandex Cloud, не Vercel

**Решение:** прод-окружение Next.js хостим в Yandex Cloud (Serverless Containers или VM), а не на Vercel.

**Причины:**
1. Инфра (PostgreSQL, Object Storage) уже в Yandex Cloud, ru-central1. Vercel в США → +200-400ms latency на каждый запрос, критично для rrweb-трекера.
2. 152-ФЗ: персональные данные граждан РФ должны обрабатываться на территории РФ. Vercel = нарушение.
3. Vercel не принимает российские карты с 2022, оплата проблематична.

**Компромисс:** сложнее настроить, нет встроенного preview deployments, нужно вручную конфигурировать CI/CD. Но для SaaS с российскими клиентами это правильный выбор.

**Альтернативы, которые рассматривали:** Vercel (отклонено по причинам выше), self-hosted на VPS (отклонено — слишком много DevOps-работы для MVP).

## 2026-04-22: Таймлайн MVP = 4-6 недель, не 1 неделя

**Решение:** MVP готовим за 4-6 недель, а не за неделю как изначально планировали.

**Причина:** Этап 0 (настройка инфры — самая простая часть проекта) занял заметно больше времени чем ожидалось. Дальше будет сложнее: Prisma + схема БД, NextAuth, rrweb-трекер с аплоадом сессий в S3, AI-анализ через Claude API, биллинг через YooKassa, email через Resend — каждый пункт это 1-3 дня минимум.

**Последствия:** не обещаем клиентам/партнёрам MVP раньше чем через 5 недель. Не выгораем, работаем устойчиво.
