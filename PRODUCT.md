# Вебмонитор — Product Definition

> Источник правды по продукту. Обновляется при каждом значимом изменении концепции.

---

## Суть продукта

Вебмонитор — SaaS-сервис для владельцев сайтов. Трекинг-скрипт записывает поведение посетителей, Яндекс.Метрика подтягивает ключевые метрики, а Claude AI анализирует данные и выдаёт конкретные рекомендации для роста конверсии, снижения отказов и увеличения времени на сайте.

**Домен**: вебмонитор.рф

---

## Стратегия по платформам

**Первая итерация — фокус на Тильде.**
Начальная аудитория привлекается через базу дизайнеров и верстальщиков nolim.cc (~4 500 человек). Они работают с Тильдой, поэтому онбординг и инструкция по установке скрипта написаны для Тильды.

**Архитектура — универсальная с первого дня.**
Трекинг-скрипт работает на любом сайте: Тильда, WordPress, Битрикс, кастомные сайты, React-приложения. Скрипт — одна строка кода, которую можно вставить в любой HTML. Технических ограничений нет.

**Следующие итерации** — инструкции по установке и кейсы для других платформ.

## Типы пользователей

### Подрядчик (дизайнер, верстальщик, маркетолог)
- Подаёт заявку «Стать партнёром», после ручной валидации админом получает доступ к кабинету
- Добавляет клиентов (владельцев сайтов) в свой кабинет через персональный invite-токен
- Получает **50% комиссии** с каждой оплаты своего клиента **пожизненно** (пока клиент платит)
- Видит список клиентов, их статусы, свой заработок и может вывести средства
- При регистрации видит **демо-стенд** с примером заполненного дашборда

### Владелец сайта
- Может зарегистрироваться **самостоятельно** на сайте и оплатить подписку без приглашения
- Либо получает персональный invite от подрядчика (email со ссылкой + временный пароль, меняет при первом входе)
- Клиент считается «приведённым подрядчиком» только если зарегистрировался по персональному invite-токену. Задним числом привязать нельзя (защита от фрода)
- Видит метрики своего сайта, рекомендации AI и аналитику динамики

### Админ (ADMIN)
- Сотрудник Вебмонитора
- Валидирует заявки подрядчиков на партнёрство
- Обрабатывает запросы на вывод средств
- Доступ к служебной панели

## Процесс приглашения клиента (опциональный путь)

Этот путь используется, когда подрядчик хочет привязать клиента к себе и получать 50% комиссии. Самостоятельная регистрация владельца — альтернативный путь без подрядчика.

1. Подрядчик заполняет короткую форму: URL сайта клиента + email клиента
2. Система создаёт InviteToken, отправляет клиенту email со ссылкой и временным паролем
3. Клиент переходит по ссылке, меняет пароль, заполняет свой профиль сам (название компании, телефон, ID Яндекс.Метрики, контакты)
4. Создаётся User (role=OWNER) и OwnerProfile с привязкой к contractorId из InviteToken
5. Подрядчик видит клиента в своём списке со статусом «Активен»

## Ключевые фичи MVP

### 1. Трекинг-скрипт
- Одна строка кода — вставляется в `<head>` любого сайта
- Базируется на **rrweb** (запись сессий: клики, скролл, движения мыши)
- Скрипт пишет **все страницы сайта** (не ограничивается списком целей анализа). Это нужно для общей статистики и накопления сессий
- Хранение данных сессий — **30 дней**, затем автоудаление
- Маскировка ввода в поля `password`, `email`, `tel` (обязательно, до отправки)
- IP-адреса хранятся только в **хэшированном виде** (требование 152-ФЗ)
- В MVP: инструкция по установке написана для Тильды

### 2. Интеграция с Яндекс.Метрикой
- Подключение через API-ключ (вводится вручную в настройках)
- Подтягиваем: визиты, уникальные посетители, цели, конверсия, отказы, время на сайте
- Ежедневная синхронизация, кнопка «Обновить сейчас»

### 3. AI-анализ (Claude API)
- Запускается вручную — **владельцем сайта или подрядчиком** (когда подрядчик открыл дашборд клиента)
- Лимиты считаются по тарифу клиента (владельца), независимо от того кто нажал кнопку
- При запуске анализа пользователь выбирает одну из заранее настроенных **целей анализа** (URL страницы)
- Анализирует записанные сессии посетителей по выбранной цели + метрики Яндекс.Метрики
- Возвращает **10–20 рекомендаций**, отсортированных по приоритету:
  - 🔴 Критично
  - 🟡 Важно
  - 🟢 Хорошо (уже работает)
- В интерфейсе по умолчанию показываются **топ-10** — остальные скрыты под кнопкой «Показать ещё N рекомендаций»
- В базе данных хранятся **все** рекомендации
- **Условия запуска нового анализа** (все должны выполняться):
  - Цель анализа добавлена в список целей сайта (нельзя запустить анализ по случайному URL — только по заранее настроенному)
  - Все топ-10 рекомендаций предыдущего анализа **этой же цели** в статусе DONE или REJECTED. Рекомендации в статусе NEW или IN_PROGRESS блокируют новый анализ этой цели. Анализы других целей не блокируют.
  - Прошло не менее **N дней** с предыдущего анализа этой цели:
    - N = 7 для тарифов Базовый и Стандартный
    - N = 3 для тарифов Профессиональный и Корпоративный
  - Не превышен месячный лимит анализов **по сайту в целом** (Базовый — 4, Стандартный — 12, Профессиональный — 24, Корпоративный — 48). Счётчик сбрасывается в начале каждого платёжного периода.

  Если хотя бы одно условие не выполнено — кнопка «Запустить анализ» заблокирована с пояснением что именно нужно сделать (например: «Обработайте 3 оставшиеся рекомендации по этой цели» или «Следующий анализ цели /pricing будет доступен 2026-05-04»).
- История всех анализов сохраняется

### 4. Статусы рекомендаций
Каждая рекомендация проходит жизненный цикл:
- **NEW** — только что появилась после анализа
- **IN_PROGRESS** — владелец нажал «Принять», взял в работу
- **DONE** — владелец нажал «Выполнено» (правка реально внесена на сайте)
- **REJECTED** — владелец нажал «Отказаться»; опционально вводит причину отказа

Все статусы видны в интерфейсе — прозрачная картина что принято, что в работе, что сделано, от чего отказались и почему.

### 5. Управление целями анализа

**Цель анализа** — конкретный URL страницы сайта, по которой можно запускать AI-анализ. Владелец настраивает список целей в кабинете.

**Что можно делать:**
- Добавить цель: указать URL (например, `https://site.ru/pricing`) и опционально дать ей человекочитаемое имя («Страница цен»)
- Удалить/архивировать цель: если больше не нужна. История анализов по этой цели сохраняется в архиве.
- Количество активных целей ограничено тарифом (см. таблицу тарифов)

**Зачем это нужно:**
- Владелец интернет-магазина может анализировать главную, корзину, страницу товара по отдельности — на каждой странице свои паттерны поведения посетителей
- Можно крутить разные страницы параллельно: пока ждёшь кулдаун по `/pricing` (7 дней), запускаешь анализ по `/about`
- При изменении рекламной кампании можно чаще анализировать конкретные посадочные страницы

**Чего НЕ происходит:**
- Трекинг-скрипт пишет ВСЕ страницы сайта (не ограничивается списком целей). Цели — это только про AI-анализ, не про трекинг
- Нельзя запустить анализ по URL, не добавленному в список целей. Если хочешь проанализировать новую страницу — сначала добавь её как цель

### 6. Дашборд владельца
- Текущие метрики сайта (конверсия, отказы, время на сайте, визиты)
- Список рекомендаций с фильтрацией по приоритету и статусу
- **Сравнение по произвольным периодам** (подробнее ниже)
- Эффект от каждой выполненной рекомендации

### 7. Аналитика динамики — сравнение по периодам
В дашборде: выбор произвольного периода для сравнения метрик «до и после».

Доступные варианты:
- «С последних правок» (с даты последней рекомендации со статусом DONE)
- «С выбранной даты» (date picker)
- «За последний месяц / 3 месяца / год»
- «Между двумя произвольными датами»

Система ежедневно сохраняет снапшот всех метрик сайта (MetricsSnapshot) и может сравнивать значения на любые две даты.

### 8. Аналитика по каждой рекомендации
Для каждой рекомендации со статусом DONE система показывает её собственный эффект:
- Мини-блок «до/после» прямо на карточке рекомендации
- «До» = снапшот метрик на момент принятия рекомендации (IN_PROGRESS)
- «После» = снапшот метрик через 7 дней после выполнения (DONE)
- Общая таблица «Эффект по рекомендациям» — сортировка по наибольшему росту

### 9. Кабинет подрядчика
- **Демо-стенд**: демо-проект «Демо-стенд Вебмонитор» с заполненным дашбордом автоматически создаётся у подрядчиков и у самостоятельно зарегистрированных владельцев. У клиентов, приведённых подрядчиком по invite, демо-стенд не создаётся (подрядчик уже показал им продукт через свой демо перед приглашением). В кабинете подрядчика демо виден в списке клиентов. Кнопка «Скопировать ссылку на демо» → подрядчик отправляет ссылку потенциальному клиенту, тот видит продукт без регистрации.
- Список реальных клиентов и их статусы
- Сводка заработка (начислено, выплачено, доступно к выводу)
- Форма добавления нового клиента (приглашение)
- Запрос на вывод средств

### 10. Дашборд клиента в кабинете подрядчика
Эта секция касается **только клиентов, приведённых подрядчиком** по invite-токену. Самостоятельно зарегистрированные владельцы в кабинет подрядчика не попадают.

Когда подрядчик кликает на клиента в своём списке — открывается дашборд этого клиента с двумя вкладками:

**Вкладка «Вид владельца»**
Подрядчик видит ровно то, что видит сам владелец: метрики, рекомендации, аналитика периодов, история событий. Позволяет объяснять клиенту что происходит и запускать анализ от его имени.

**Вкладка «Инструменты подрядчика»** *(только для подрядчика, владелец её не видит)*
- Статус доступа клиента (заходил ли, когда последний раз)
- Готовый код скрипта для установки (с токеном клиента, один клик — скопировать)
- Статус интеграции с Яндекс.Метрикой (подключена / не подключена, ошибки)
- Быстрые действия: «Написать клиенту», «Отправить PDF-отчёт», «Открыть сайт клиента»

### 11. Контакт с поддержкой и подрядчиком (в дашборде владельца)
В шапке дашборда владельца всегда отображается блок с контактом **саппорта Вебмонитора** — базовая поддержка для любого владельца.

Если владелец **приведён подрядчиком**, дополнительно отображается блок с подрядчиком:
- Аватар и имя подрядчика
- Кнопка «Написать» → открывает модальное окно с вариантами связи: WhatsApp, Telegram, Email
- Контактные данные подрядчик вводит при регистрации (телефон, Telegram-ник, email)

У **самостоятельно зарегистрированных** владельцев блок с подрядчиком не отображается — только саппорт.

Цель: владелец всегда знает к кому обратиться с вопросом — к саппорту Вебмонитора или к своему подрядчику (если он есть).

### 12. История событий (в дашборде владельца)
Хронологическая лента всех значимых событий по сайту:

- Зарегистрировался самостоятельно
- Зарегистрировался по приглашению от подрядчика
- Подключён счётчик Яндекс.Метрики
- Установлен трекинг-скрипт (первая сессия)
- Добавлена цель анализа (URL)
- Архивирована цель анализа (URL)
- Запущен AI-анализ (цель, дата, количество рекомендаций)
- Рекомендация взята в работу (название)
- Рекомендация выполнена (название)
- Рекомендация отклонена (название, причина)
- Изменены настройки сайта
- Оплата тарифа (план, период)

Каждая запись: дата + короткое описание + иконка типа события. Это не дублирует аналитику по рекомендациям — это именно лог событий для понимания «что вообще происходило с сайтом».

- Годовая оплата списывается единовременно.

## Юридические требования

- **152-ФЗ**: все данные пользователей хранятся на серверах в РФ (Yandex Cloud, регион ru-central1)
- **Маскировка PII**: поля `password`, `email`, `tel` маскируются на стороне клиентского скрипта до отправки на сервер
- **IP-адреса**: хэшируются (SHA-256 + соль) до записи в базу, оригинал не сохраняется
- **Юрлицо**: ООО «Супер Просто» — используется для подключения ЮКассы и всех коммерческих операций
- Необходимо пользовательское соглашение и политика конфиденциальности

## Рынок и запуск

- Первый рынок — **Россия** (русскоязычный интерфейс, рублёвые тарифы)
- **Основной канал — self-service воронка**: контент-маркетинг, SEO, реклама, сарафан — владельцы сайтов регистрируются и оплачивают самостоятельно
- **Дополнительный канал — партнёрская программа**: база **nolim.cc** (~4 500 дизайнеров и верстальщиков Тильды), клиенты приходят через валидированных подрядчиков
- Домен: **вебмонитор.рф** (кириллический, требует Punycode в технических контекстах)
