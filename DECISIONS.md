## 2026-04-22 — Деплой — Yandex Cloud, не Vercel

**Решение:** прод-окружение Next.js хостим в Yandex Cloud (Serverless Containers или VM), а не на Vercel.

**Причины:**
1. Инфра (PostgreSQL, Object Storage) уже в Yandex Cloud, ru-central1. Vercel в США → +200-400ms latency на каждый запрос, критично для rrweb-трекера.

## 2026-04-22: Workflow миграций Prisma для Yandex Managed PostgreSQL

**Проблема:** `npm run prisma migrate dev` не работает с Yandex Managed PostgreSQL — у пользователя webmonitor нет прав на CREATE DATABASE для shadow database (стандартный механизм Prisma для сравнения схемы в dev-окружении).

**Решение — правильный workflow миграций:**

1. Изменить prisma/schema.prisma (добавить модель, поле, enum и т.д.)

2. Сгенерировать SQL миграцию БЕЗ shadow DB:
   ```
   npm run prisma migrate diff \
     --from-migrations prisma/migrations \
     --to-schema-datamodel prisma/schema.prisma \
     --script > /tmp/migration.sql
   ```
   Проверить содержимое /tmp/migration.sql глазами — не должно быть DROP TABLE без явного намерения, подозрительных ALTER и т.п.

3. Создать папку миграции вручную:
   ```
   mkdir -p prisma/migrations/<timestamp>_<descriptive_name>
   mv /tmp/migration.sql prisma/migrations/<timestamp>_<name>/migration.sql
   ```
   Где `<timestamp>` — формат YYYYMMDDHHMMSS, `<descriptive_name>` — snake_case, например `add_site_model` или `user_email_verified_field`.

4. Применить миграцию:
   ```
   npm run prisma migrate deploy
   ```

5. Обновить Prisma Client:
   ```
   npm run prisma generate
   ```

6. Проверить что всё в синхроне:
   ```
   npm run prisma migrate status
   ```
   → должно вернуть "Database schema is up to date!"

**Альтернатива на будущее (когда появится локальная БД):** настроить shadow database через Docker Postgres и указать shadowDatabaseUrl в prisma.config.ts — тогда `migrate dev` заработает как обычно.

**Применённые миграции:**
- `20260422000000_init_user_model` — первая модель User с enum UserRole (ADMIN, CONTRACTOR, OWNER)

## 2026-04-23: Модель привлечения — self-service для владельцев + партнёрская программа для подрядчиков

**Решение:** меняем модель с "closed beta через подрядчиков" на open self-service с опциональной партнёркой.

**Как работает:**
- Владелец сайта может САМ зарегистрироваться и оплатить подписку без приглашения.
- Подрядчик (например, дизайнер Тильды) подаёт заявку "Стать партнёром" → админ валидирует вручную → у подрядчика появляется кабинет с функцией приглашения клиентов.
- Клиент считается "приведённым подрядчиком" ТОЛЬКО если зарегистрировался по персональному invite-токену от этого подрядчика.
- Самостоятельно зарегистрированных клиентов привязать к подрядчику задним числом НЕЛЬЗЯ (защита от фрода).

**Причина изменения:** касдев с владельцами сайтов показал что они хотят самостоятельной регистрации, а не "приглашения от дизайнера".

**Последствия для архитектуры:**
- Нужны публичные страницы регистрации и оплаты для владельцев.
- Партнёрская программа — отдельный раздел с заявками и валидацией.
- Модель InviteToken остаётся, но теперь это дополнительный (не единственный) путь попадания в систему.

## 2026-04-23: Комиссия подрядчику — 50% навсегда, фикс

**Решение:** подрядчик, приведший клиента по своему invite-токену, получает 50% от каждого платежа клиента ПОЖИЗНЕННО (пока клиент платит).

**Альтернативы которые рассмотрели и отклонили:**
- Первый год 50%, потом 10% → сложнее считать, мотивация партнёра падает резко.
- Только первый год → короткая мотивация, подрядчики не будут поддерживать клиентов долгосрочно.
- Разный % для разных партнёров → слишком сложно для MVP, отложим.

**Причина:** простота для MVP, ясная мотивация партнёрам, легко считать.

**На будущее:** можно будет ввести разные % по сегментам (например, топовым партнёрам давать больше), но не сейчас.

## 2026-04-24: Условия запуска AI-анализа — обработка + кулдаун 7 дней

> **ЧАСТИЧНО УСТАРЕЛО 2026-04-27** — общая идея «обработка + кулдаун + лимит» сохраняется, но конкретные цифры и логика изменились: добавлено условие «цель в списке», кулдаун работает на пару (сайт, цель) а не глобально, длительность кулдауна зависит от тарифа (7 или 3 дня), лимит анализов по тарифам стал 4/12/24/48. См. запись «2026-04-27: Тарифная сетка переработана» ниже.

**Решение:** для запуска нового AI-анализа должны одновременно выполняться три условия:

1. Все рекомендации из топ-10 предыдущего анализа переведены в статус DONE или REJECTED (не NEW, не IN_PROGRESS)
2. С момента предыдущего анализа прошло минимум 7 дней
3. Не превышен месячный лимит по тарифу (Basic 1 / Pro 2 / Business 4)

**Обоснование:**

Условие 1 защищает от хаотичного использования:
- без него юзер запускает анализ, получает 15 рекомендаций, ничего не делает, завтра снова запускает — и у него 30 «висящих» карточек
- это убивает UX и обесценивает AI-рекомендации
- статус IN_PROGRESS считается «необработанным» — юзер должен явно довести до DONE или REJECTED

Условие 2 (7 дней) — продуктовое и экономическое обоснование:
- за 24 часа метрики не успевают измениться, вся аналитика metricsBefore/metricsAfter будет бессмысленной
- поле metricsAfter в схеме БД уже берётся «через 7 дней после DONE» — 7-дневный кулдаун синхронизируется с этим
- даёт юзеру время реально внедрить изменения, а не нажимать «Запустить» в спешке

Условие 3 — экономика тарифов:
- себестоимость одного анализа через Claude API ≈ 15-30₽, поэтому лимит важен для маржинальности
- цифры 1/2/4 дают маржу >95% на всех тарифах

**Важный нюанс для Business-тарифа:**

4 анализа в месяц / 7 дней кулдаун = теоретически ~4 анализа возможны, но сжато. Если юзер быстро обрабатывает рекомендации — успеет все 4. Если медленно — получит 2-3. Это **фича, не баг**: лимит — «максимум», фактическое количество зависит от темпа использования.

**UI-поведение:** кнопка «Запустить анализ» заблокирована если не выполнено хотя бы одно условие. Показываем пояснение — какое именно условие и что надо сделать:
- «Обработайте N оставшихся рекомендаций»
- «Следующий анализ будет доступен DD.MM.YYYY» (по кулдауну или по лимиту — в зависимости от того что сработает первым)

## 2026-04-25: Архитектура — модульный монолит, не микросервисы

**Решение:** MVP и ранняя стадия пилятся как модульный монолит (один Next.js проект с чётким разделением на модули), не микросервисы.

**Обоснование:**
1. У нас один разработчик (продакт + Claude Code) — микросервисы оправданы когда толпятся несколько команд.
2. Операционные расходы микросервисов (отдельные деплои, API gateway, service mesh, межсервисные логи) для MVP — гарантированное самоубийство.
3. Массовый тренд 2025-2026: значительная доля компаний, принявших микросервисы, возвращают часть функций обратно в монолит из-за сложности отладки и операционной нагрузки (см. отчёты CNCF, статьи в InfoQ, DZone). Конкретные цифры варьируются от источника к источнику — тренд важнее цифр.
4. AI-ассистенты (Claude Code) эффективнее работают с монорепо — весь контекст в одном месте.
5. Сетевая латентность между сервисами (10-50ms на вызов) критична для rrweb-трекера, который шлёт данные каждые 30 сек.

**Модульность обеспечиваем структурой** (точные пути см. в ARCHITECTURE.md → "Структура репозитория"):
- Отдельные зоны UI под роли: `app/(dashboard)` для владельцев, `app/(contractor)` для подрядчиков, `app/(admin)` для админки
- Отдельные группы API-эндпоинтов по функциональным областям: auth, tracking, metrika, analysis, webhooks (ЮКасса), clients
- Вспомогательные модули в `lib/`: prisma, claude, metrika, yookassa, resend, storage, crypto — по одному на внешнюю интеграцию или кросс-проектную утилиту
- Межмодульные вызовы — только через чёткие интерфейсы (функции, типы), без прямых обращений в приватные куски чужого модуля

**Когда пересмотреть:** если у нас 5+ разработчиков ИЛИ 100k+ строк кода ИЛИ один компонент стал узким местом по производительности.

**Альтернативы которые рассмотрели:**
- Микросервисы с первого дня — отклонено по причинам выше
- Чистый монолит без модульности — отклонено, это путь к спагетти-коду через полгода

## 2026-04-27: «Цель анализа» как сущность — модель AnalysisTarget

**Решение:** вводим в схему БД новую таблицу AnalysisTarget — это URL страницы сайта, по которой пользователь может запускать AI-анализ. Каждый Analysis привязан к конкретному AnalysisTarget через targetId.

**Логика:**
- Цель = один URL (например, `https://site.ru/pricing`). Не паттерн, не группа URL — на MVP только конкретные URL.
- Цели создаёт владелец сайта в кабинете. Лимит активных целей зависит от тарифа.
- Кулдаун между анализами считается на пару (siteId, targetId), не глобально по сайту.
- Условие «обработать топ-10» применяется к последнему анализу этой же цели — анализы других целей не блокируют.
- Лимит анализов в месяц — суммарный по сайту (любые цели в любой комбинации).
- При архивации цели (`archivedAt`) она исчезает из UI выбора при запуске анализа, но прошлые Analysis остаются доступны в истории.

**Обоснование выбора «один URL» а не «паттерн URL»:**
- Проще для MVP (нет ambiguity «совпадает ли /products/123 с /products/*»).
- Понятнее в UI (юзер указывает конкретный URL, видит конкретный URL).
- Можно добавить паттерны позже если будет запрос (особенно от интернет-магазинов с тысячами товарных страниц).

**Технические детали:**
- AnalysisTarget.url — string, без уникальности (юзер может теоретически добавить дубль; обработка дублей — на уровне UI с предупреждением).
- AnalysisTarget.archivedAt nullable — soft delete, потому что Analysis.targetId ссылается на цель и история должна сохраняться.
- Analysis.targetId NOT NULL — каждый новый анализ обязательно привязан к цели; legacy-данных нет (модель вводится одновременно с AnalysisTarget на этапе 0.c.5).

**Связанные изменения:**
- ARCHITECTURE.md (коммит bd2c6da): схема модели AnalysisTarget, изменения в Analysis, секция «Условия запуска нового анализа».

## 2026-04-27: Модель Subscription и SubscriptionEvent — решения по этапу 0.c.4

**Решение:** реализуем биллинговый слой через две таблицы — `Subscription` (одна запись на Owner) и `SubscriptionEvent` (immutable лог всех изменений подписки, не только финансовых).

### Subscription — одна запись на Owner за всю историю

`Subscription.ownerId` объявлен `@unique`. При апгрейде/даунгрейде/продлении меняем поля той же записи, а не создаём новую. История тарифов восстанавливается через `SubscriptionEvent`.

**Альтернатива которую отклонили:** хранить историю тарифов как series записей в Subscription с полями `validFrom`/`validTo`. Отклонено, потому что:
- При каждом обращении к "текущей подписке" пришлось бы делать `WHERE validTo IS NULL` — лишняя сложность
- История уже хранится в SubscriptionEvent — дублирование

### SubscriptionEvent, а не PaymentEvent

Логируем не только денежные события, но и любые изменения подписки: апгрейды, даунгрейды, отмены через UI без платежа, истечение периода. Без этого нельзя восстановить полную историю тарифов клиента — например, "юзер отменил подписку" без платежа не оставит следа.

**Технически:** поля `amount`, `currency`, `yookassaPaymentId` сделаны nullable — для не-денежных событий они пустые. Для бухгалтерской выгрузки фильтр `WHERE amount IS NOT NULL` отдаёт только платежи.

`yookassaPaymentId` — `@unique`, защита от двойной обработки вебхука.

### analysesUsedThisPeriod при смене тарифа НЕ сбрасываем

При апгрейде/даунгрейде счётчик использованных анализов в текущем периоде остаётся как есть — защита от абьюза по схеме BASIC→STANDARD→BASIC (юзер мог бы за месяц получить 4+12+4 = 20 анализов вместо 4 на Базовом).

**Последствие:** если юзер потратил 8 анализов на STANDARD и сделал даунгрейд на BASIC (лимит 4) — анализы заблокированы до начала следующего платёжного периода. Это корректное поведение, не баг. Логика проверки лимита — в коде, не в схеме.

### Период биллинга — от даты подписки

Подписался 14-го числа → следующий период начинается 14-го следующего месяца. Не календарный месяц. Расчёт через `date-fns` (`addMonths`), который корректно обрабатывает кейсы типа "31 января → 28/29 февраля".

Поля `currentPeriodStart` и `currentPeriodEnd` хранятся в Subscription, обновляются при каждом успешном продлении. Сброс `analysesUsedThisPeriod` происходит только при создании события `PAYMENT_SUCCEEDED` типа RENEWAL.

### Триал на MVP не делаем

В enum `SubscriptionStatus` нет TRIAL. Когда понадобится — добавим миграцией.

### Enterprise тариф на MVP не делаем

В enum `SubscriptionTier` нет ENTERPRISE и в `Subscription` нет поля `customLimits Json?`, хотя это упоминается в PRODUCT.md и в ARCHITECTURE.md как стратегическое направление. Причина: Enterprise — ручной B2B-процесс с переговорами, на MVP клиентов не будет. Когда появится первый — добавим миграцией ENTERPRISE в enum и поле customLimits.

**Альтернатива на MVP:** если кому-то нужны нестандартные лимиты — устанавливаем руками через Prisma Studio (например, `analysesLimit = 999` для конкретного Subscription).

## 2026-04-27: Модель «Бюджеты сессий на цель» — переход с кулдауна на распределение лимита сессий

**Решение:** меняем модель регулирования AI-анализов. Раньше частоту запусков ограничивали два механизма — кулдаун в днях (7 для BASIC/STANDARD, 3 для PROFESSIONAL/CORPORATE) и месячный лимит анализов. Теперь оставляем только месячный лимит анализов (4/12/24/48), а вместо кулдауна вводим **бюджеты сессий на каждую цель** — юзер сам распределяет общий лимит сессий тарифа между AnalysisTarget'ами, и анализ запускается когда у конкретной цели накоплено ≥100 сессий.

### Что было не так с предыдущей моделью

В исходной концепции (см. записи от 2026-04-23 и 2026-04-27 про тарифную сетку) лимит сессий был «общим по сайту» — просто счётчиком сессий за месяц. А кулдаун в днях защищал от спама анализов. Эта модель имела две проблемы:

1. **Низкотрафичные страницы получали мусорный анализ.** Если на странице «О компании» 30 сессий в неделю, по правилу «прошло 7 дней — можно анализировать» Claude получал бы 30 сессий и выдавал бы галлюцинации. Качество рекомендаций не масштабировалось.

2. **Стоимость одного анализа была непредсказуемой.** Юзер на BASIC мог запустить анализ на 1000 сессиях главной страницы — расход токенов Claude варьировался в разы между анализами. Экономика тарифа плыла.

3. **Кулдаун был эвристикой**, не отражающей реальной готовности страницы к новому анализу. На высокотрафичной главной за 7 дней могло набраться 5000 новых сессий — кулдаун избыточен. На низкотрафичной — за 7 дней набралось 50 сессий, но по таймеру можно — и анализ выходит мусорный.

### Новая модель: бюджет сессий на цель

Юзер при создании AnalysisTarget указывает `sessionsBudget` — потолок сессий, которые цель должна накопить перед анализом. Минимум 100 (защита от мусорных анализов), максимум — оставшийся свободный баланс тарифа. Сумма бюджетов всех активных целей не может превышать `Subscription.sessionsLimit`.

Скрипт собирает сессии: при закрытии каждой сессии сервер определяет на какой странице посетитель провёл больше всего времени; если страница совпадает с url одной из активных целей сайта — счётчик `sessionsCollected` этой цели увеличивается на 1. Когда `sessionsCollected ≥ 100` — цель в статусе `READY`, кнопка «Запустить анализ» активна.

После запуска анализа цель переходит в `ANALYZING` → `COMPLETED`. В текущем периоде повторный анализ этой же цели невозможен. Сброс происходит при обновлении периода Subscription (счётчики обнуляются, бюджеты сохраняются).

### Принятые решения по поведению модели

Все решения зафиксированы по результатам опросника:

1. **Сессия посетителя засчитывается ровно одной цели** — той, на странице которой он провёл больше всего времени за сессию. Не «первой посещённой», не «последней» — а максимальной по длительности. Технически: при `endedAt` сервер анализирует rrweb-таймлайн URL, выбирает максимум среди тех URL, что есть в активных целях сайта (status ∈ {ACTIVE, READY}).

2. **Минимум 100 сессий** — это и минимальный `sessionsBudget` (нельзя поставить цель с потолком меньше 100), и минимум для запуска анализа (`sessionsCollected ≥ 100`).

3. **Сумма бюджетов не может превышать `sessionsLimit` тарифа** — UI блокирует сохранение, если сумма аллокаций больше лимита. Защита от ситуации «накопилось 2500 сессий, но они «забронированы» под 3000 сессий бюджета».

4. **Параллельные анализы разрешены** — юзер может запустить анализ нескольких целей одновременно (если у каждой накоплено ≥100). Ограничение только по общему лимиту анализов в месяц.

5. **Архивация цели возвращает бюджет на свободный баланс**, если по этой цели ещё не запускался анализ. Если анализ был — бюджет уже потрачен и не возвращается. Накопленные `sessionsCollected` при архивации сгорают (счётчик зануляется).

6. **«Отменить сбор оставшихся» после запуска анализа** — кнопка появляется сразу при переходе цели в ANALYZING (даже пока Claude ещё думает). Юзер может нажать — `sessionsBudget` цели обрезается до фактического `sessionsCollected` на момент анализа, разница возвращается на свободный баланс.

7. **При обновлении периода (платёжного):** счётчики `sessionsCollected` всех целей обнуляются, цели в статусе COMPLETED → ACTIVE (можно повторно), бюджеты сохраняются как были. Цели в ARCHIVED остаются как есть.

8. **Если подписка истекла (юзер не оплатил):** трекинг-скрипт перестаёт принимать сессии — `/api/tracking/collect` возвращает 402 (или 403). Старые анализы и накопленные данные доступны read-only. При новом платеже всё разморозится с нуля.

9. **Месячный лимит анализов остаётся** (4/12/24/48). Это защита экономики Claude — один анализ это реальные деньги в облаке. Хотя на практике лимит сессий и так ограничивает кейсы (если 1000 сессий распределены по 4 целям по 250, то будет ≤4 анализа), он режет редкий граничный кейс — много мелких целей по 100 сессий.

### Что изменилось в схеме БД (этап 0.c.5)

- `Subscription`: новые поля `sessionsLimit Int` и `sessionsAllocated Int`. Денормализованный счётчик аллоцированных сессий — для быстрых проверок UI без агрегата при каждом запросе.
- `AnalysisTarget`: новые поля `sessionsBudget Int` (минимум 100), `sessionsCollected Int @default(0)`, `status AnalysisTargetStatus`. Новый enum `AnalysisTargetStatus { ACTIVE | READY | ANALYZING | COMPLETED | ARCHIVED }`.
- `Session`: новое поле `analysisTargetId String?` — FK на AnalysisTarget с `ON DELETE SET NULL`. Проставляется при закрытии сессии.

### Что отменяется

- **Кулдаун в днях между анализами одной цели** — больше не используется. Запись «2026-04-27: Тарифная сетка переработана» помечена как частично отменённая в этой части.
- **Поле `Subscription.sessionsLimit` в значении «общий счётчик сессий за месяц»** — теперь это «общий бюджет на распределение между целями». Семантика поменялась.

### Альтернативы которые рассмотрели и отклонили

- **«Сессия засчитывается всем посещённым целям»** — отклонено: одна сессия = одна единица бюджета, иначе бюджеты теряют смысл (одна сессия посетителя могла бы тратить 3 единицы из разных целей).
- **«Засчитывается первой/последней посещённой»** — технически проще, но продуктово менее честно. Юзер мог зайти на /pricing, потом случайно кликнуть на /about; считать сессию за /about неправильно.
- **`sessionsAllocated` высчитывать на лету через SUM(sessionsBudget)** — отклонено: считается на каждом запросе UI (проверка «можно ли создать ещё одну цель»), денормализованный кэш быстрее. Поддерживаем актуальность через триггеры в коде.
- **Кулдаун оставить + добавить бюджеты сверху** — отклонено: два механизма блокировки запуска избыточны и запутывают юзера.
